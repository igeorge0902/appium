Appium Test Suite System Documentation — March 2026

Automated UI tests for the SwiftCinemas iOS app using Appium 2 + XCUITest driver + TestNG. The test suite exercises the full booking journey: app launch → movie browse → category filter → venue select → date pick → seat reserve → checkout.

Framework

Appium 9.3 Java Client, Selenium 4.27, TestNG 7.10, SLF4J + Logback. Java 17.

Patterns

Page Object Model with static navigation methods, custom TestNG listeners, screenshot-on-failure.

Platform

iOS Simulator (primary) and real device via ideviceinfo auto-detection. Appium 3 / XCUITest driver.

⚠️ Backend Dependency

The test flow navigates Movies → Categories → Venues → Seats → Checkout. Every screen after the home page fetches data from the backend API. Without a running Cinemas system (MySQL, Kafka, mbooks, dalogin, etc.), the Movies table is empty, the segmented control never renders, and all tests fail with NoSuchElement. See Backend Requirements.

Class Hierarchy

The framework uses a deep inheritance chain with a mix of inheritance and static method patterns. All navigation and page object classes extend TestBase, which provides access to the shared IOSDriver instance and verification utilities.

Key Design Decisions

  • Static driver reference: TestBase.driver and DriverManager.driver are static fields. All navigation/page classes access the driver via inheritance. This means only one Appium session can run at a time per JVM.
  • Static navigation methods: All navigation classes use public static methods that return new ClassName(). The return values are unused — this is a builder-pattern remnant from earlier iterations.
  • Verify vs Assert: The Verify class provides soft assertions — failures are collected per-test-method via TestMethodErrorBuffer and reported at the end by TestMethodListener, allowing tests to continue after a verification failure.

Driver Lifecycle

1
TestBase constructor — calls DeviceConfiguration.getDevices() which runs ideviceinfo via CommandPrompt. If a real device is connected, deviceCount > 0. If not (or ideviceinfo is not installed), deviceCount = 0 (simulator mode).
2
@BeforeClass setUp() — reads app and device parameters from testng.xml. Calls DriverManager.startDriver(app, device, timeout).
3
DriverManager.startDriver() — builds XCUITestOptions based on deviceCount:
  • Real device (deviceCount > 0): uses UDID, ProductVersion, xcodeOrgId, signing identity from ideviceinfo output.
  • Simulator (deviceCount == 0): reads sim.device, sim.version, sim.udid from system properties; resolves the .app path relative to user.dir.
Constructs the Appium server URL from appium.host/appium.port/appium.basePath. Creates IOSDriver. Sets 30s WebDriverWait, 30s page load, 60s implicit wait, snapshot timeout 600s, max depth 200.
4
App reset — after driver creation, setUp() calls terminateApp(bundleId) + activateApp(bundleId) to ensure a clean app state (replaces deprecated resetApp()).
5
Tests execute — navigation classes call driver.findElement(), driver.executeScript() etc.
6
@AfterClass closeBrowser() — calls DriverManager.stopDriver() which terminateApp(bundleId) + driver.quit().

Retry on First Failure

setUp() wraps the initial startDriver() call in a try-catch. If the first attempt fails (e.g., WDA build timeout, network issue), it retries exactly once with the same parameters. This handles transient WDA startup failures.

TestNG Listeners

Listener Registration

Listeners are registered via @Listeners annotation on test classes:

@Listeners({TestListeners.class, CaptureScreenshotOnFailureListener.class,
             TestMethodListener.class, LoggingListener.class})
public class TestNavigations extends TestBase { ... }

TestListeners

Implements ITestListener, ISuiteListener, IInvokedMethodListener. Logs lifecycle events (suite start/end, test start/pass/fail/skip) via Reporter.log() and SLF4J. Calls printTestResults() on each test outcome to report the class name, parameters, and status.

TestMethodListener

Implements IInvokedMethodListener. Manages the TestMethodErrorBuffer lifecycle: initialises a fresh ArrayList<Throwable> before each test method, and after each test method aggregates any soft-verification failures from TestBase.getVerificationFailures() into the test result. If multiple failures exist, constructs a combined failure message with stack traces.

CaptureScreenshotOnFailureListener

Extends TestListenerAdapter. On onTestFailure(), grabs a screenshot via TakesScreenshot, saves to test-output/html/ScreenShots/, and injects an HTML <img> tag into the TestNG report.

LoggingListener

Extends TestListenerAdapter. Logs pass/fail/skip events with colourful Hungarian-language commentary. On failure: logs to SLF4J. On success: "Minden fasza". On skip: "Shit happens".

Verification Error Buffer (Thread Safety)

TestMethodErrorBuffer uses a ThreadLocal<List<Throwable>> to store verification failures per thread — safe for parallel test execution. The lifecycle is:

  1. TestMethodListener.beforeInvocation() → removes stale buffer, creates fresh ArrayList
  2. During test: Verify.addToErrorBuffer() adds VerificationError instances
  3. TestMethodListener.afterInvocation() → checks TestBase.getVerificationFailures(), sets test status to FAILURE if any exist, cleans up

Verification Framework

File: src/main/java/qa/ios/testng/Verify.java

Provides soft-assertion wrappers around TestNG Assert methods. Instead of immediately failing the test on assertion error, failures are captured and collected for reporting at the end of the test method.

Available methods

MethodDelegates to
verifyTrue(condition, message)Assert.assertTrue()
verifyFalse(condition, message)Assert.assertFalse()
verifyEquals(actual, expected)Assert.assertEquals() — overloaded for Object, String, boolean, Collection, Iterator, Iterable, Object[]
verifyNotNull(object)Assert.assertNotNull()
verifyNull(object)Assert.assertNull()
verifySame(actual, expected)Assert.assertSame()
verifyNotSame(actual, expected)Assert.assertNotSame()
verifyEqualsNoOrder(actual, expected)Assert.assertEqualsNoOrder()
fail(message)Assert.fail()

Each method catches AssertionError (sic — TestNG's spelling) and delegates to addToErrorBuffer() which creates a VerificationError and pushes it onto the TestMethodErrorBuffer.

Device Detection

DeviceConfiguration.getDevices() runs ideviceinfo at startup. The decision tree:

TestBase constructor
  └── DeviceConfiguration.getDevices()
        └── CommandPrompt.runCommand("ideviceinfo")
              ├── output has >1 line → Real device detected
              │   ├── Parse DeviceName, DeviceClass, UniqueDeviceID, ProductVersion
              │   └── deviceCount = map.size() (>0)
              │         └── DriverManager uses REAL device options
              │               (xcodeOrgId, xcodeSigningId, bundleid from properties)
              │
              └── output has ≤1 line → No device / ideviceinfo not found
                  └── deviceCount = 0
                        └── DriverManager uses SIMULATOR options
                              (sim.device, sim.version, sim.udid, .app path)

Note: If ideviceinfo is not installed, CommandPrompt.runCommand() throws an IOException which is caught by TestBase's constructor, and deviceCount remains 0 (simulator mode).

WebDriverAgent (WDA)

The XCUITest driver uses WebDriverAgent to control the iOS Simulator. Understanding the WDA lifecycle is key to troubleshooting session startup failures.

Managed Lifecycle (Default)

1
Appium's XCUITest driver builds WDA from source using Xcode
2
Installs the WDA runner app on the simulator
3
Launches WDA — it starts an HTTP server inside the simulator
4
Appium discovers the host:port and proxies all commands through WDA

⚠️ Do NOT set webDriverAgentUrl for simulators

Setting webDriverAgentUrl tells Appium: "WDA is already running at this URL — don't build or start it." If nothing is listening, you get ETIMEDOUT 192.168.0.103:8100.

The DriverManager deliberately does not set webDriverAgentUrl and sets usePreinstalledWDA: false to let Appium handle the full WDA lifecycle.

WDA Build Cache

After the first successful session, WDA is cached in Xcode's DerivedData. Subsequent sessions reuse the cached build and start much faster. To clear:

# Clear WDA build cache
rm -rf ~/Library/Developer/Xcode/DerivedData/WebDriverAgent-*

# Or uninstall WDA from the simulator
xcrun simctl uninstall <UDID> com.facebook.WebDriverAgentRunner.xctrunner

Why the Segmented Control Requires Backend Data

The UISegmentedControl with "Reset/Categories/NA" is added programmatically in MoviesVC.viewForHeaderInSection (line 385 of the Swift source). Known Issues

1. Backend required for 80% of test flow

Steps 2b through 4f all require a running backend. The UISegmentedControl only renders when the movies table has data. Without the backend, all tests fail with NoSuchElement at the segmented control step.

Impact: Critical — test suite unusable without full Kubernetes stack.

2. Fragile class-chain locators

The seat popover locator in bookChair() uses exact window/element indices: **/XCUIElementTypeWindow[1]/XCUIElementTypeOther[6]/XCUIElementTypePopover/.... This breaks across iOS versions, screen sizes, and app layout changes.

Fix: Replace with **/XCUIElementTypePopover/**/XCUIElementTypeCell[2] or predicate-based locators.

3. Hardcoded stale date in picker

VenueDetails.selectPickerDate() sends "Sep 12, 2021 at 9:00 PM" to the picker wheel. This date is stale and won't match any real screening data.

Fix: Use mobile: selectPickerWheelValue to pick the first available date dynamically.

4. iOS 26 segmented control child type

On iOS 26, UISegmentedControl children may be XCUIElementTypeSegment instead of XCUIElementTypeButton. The locator **/XCUIElementTypeSegmentedControl/XCUIElementTypeButton[2] may fail.

Fix: Use AppiumBy.iOSNsPredicateString("label == 'Categories'") instead.

5. "dismiss popup" label may differ on iOS 26

The iOS accessibility label for the popover dismiss dimming view may have changed (e.g., "Dismiss pop-up").

Fix: Inspect with InspectUI and update if needed.

6. No explicit waits between navigation steps

Navigation relies on the 60s implicit wait rather than explicit WebDriverWait conditions. This makes tests slow (waits full 60s on failure) and unreliable (may proceed before animations complete).

Fix: Add WebDriverWait + ExpectedConditions before each element interaction.

7. Static mutable state in framework

TestBase.driver, DriverManager.driver, and devices are all static. This prevents parallel test execution and can cause stale references across test classes.

8. Hardcoded Apple Developer Team ID

DriverManager has hardcoded xcodeOrgId = "X8DL5FV4EA". This should be externalised to properties or environment variable.

9. CreateXml.java uses Swing

The 584-line Swing GUI for generating testng.xml is a legacy tool. The suite files are now maintained manually. Consider removing to reduce codebase complexity.

Troubleshooting

SessionNotCreatedException: Unable to start WebDriverAgent session

  • "Application is unknown to FrontBoard" — The app is not installed on the simulator. Run xcrun simctl install booted SwiftCinemas.app.
  • ETIMEDOUT on port 8100 — Remove any webDriverAgentUrl capability; let Appium manage WDA. The DriverManager already does this correctly.
  • WDA build fails — Run appium driver update xcuitest and clear the WDA cache: rm -rf ~/Library/Developer/Xcode/DerivedData/WebDriverAgent-*

NoSuchElement during test execution

  • Backend not running — Most likely cause. Verify: curl -sk https://milo.crabdance.com/mbooks-1/rest/book/movies | head -c 200
  • Different screen layout — Different device size or iOS version. Run InspectUI to check element hierarchy.
  • Animation not finished — Increase implicit wait or add explicit waits.
  • Locator changed — iOS version update changed element types. Dump page source and update locators.

Appium server not reachable

# Check if Appium is running
curl http://127.0.0.1:4723/status

# Check what's on port 4723
lsof -i :4723

Wrong architecture

The .app binary must match the simulator platform:

  • Apple Silicon (M1/M2/M3/M4) → arm64 simulator build
  • Intel Mac → x86_64 simulator build

Check: file SwiftCinemas.app/SwiftCinemas

Build with ONLY_ACTIVE_ARCH=YES to match the host machine.

First run is very slow

The first Appium session compiles WebDriverAgent from source — this can take several minutes. Subsequent runs reuse the cached build. If WDA seems stuck, check wdaLaunchTimeout (currently 240s) and wdaStartupRetries (currently 4).
